Graphes de code et contexte IA

Comment Graphify et Code Review Graph peuvent aider Codex/Copilot à mieux naviguer dans une codebase — et ce que le benchmark LORQ montre vraiment.

Verdict globalGraphify

Meilleur candidat steady-state dans ce benchmark.

Nuance cléQualité ≠ coût

Le gain observé porte surtout sur la navigation et la réponse.

À ne pas promettre- tokens garanti

Le benchmark ne démontre pas une économie automatique.

Slide 1

Pourquoi on teste ça ?

Dans une grosse codebase, l’agent IA peut consommer beaucoup de tokens parce qu’il doit retrouver lui-même les bons fichiers, symboles et relations.

Problème 1

Le contexte brut est large : fichiers, références, tests, conventions, dépendances.

Problème 2

Une grande fenêtre de contexte ne garantit pas que le bon contexte est présent.

Question

Peut-on fournir moins de bruit et plus de contexte utile à Codex/Copilot ?

Note speaker : le sujet n’est pas seulement de réduire les tokens, mais d’augmenter la densité utile du contexte.
Repo testé : nopCommerceAngle : coût input IA + qualité de réponse
Slide 2

Structurer le code avant de le donner à l’IA

L’idée est de passer du code comme texte à une carte exploitable de la codebase.

Code source Arbre syntaxique : AST / CST Symboles : classes, méthodes, fonctions, imports Relations : appelle, dépend de, hérite, teste Graphe de code Contexte ciblé pour l’agent IA
Le graphe sert de carte de départ : il oriente la recherche, mais ne remplace pas la vérification dans le code source.
Slide 3

AST, CST et Tree-sitter en 90 secondes

Tree-sitter transforme le code texte en arbre syntaxique exploitable.

L’agent peut ensuite raisonner sur des symboles et des relations, plutôt que sur une simple recherche textuelle.

Code source

public decimal Add(decimal a, decimal b)
{
    return a + b;
}

CST simplifié

method_declaration
├── modifier: public
├── return_type: decimal
├── name: Add
├── parameters
│   ├── decimal a
│   └── decimal b
└── body
    └── return_statement
        └── binary_expression: +
            ├── identifier: a
            └── identifier: b

Ce qu’on retient

On peut extraire fonctions, méthodes, classes, imports, appels, héritages ou tests liés.

Note speaker : ne pas passer plus de 90 secondes ici. Le point important n’est pas la théorie AST/CST, mais la transition “code texte → structure → graphe → contexte IA”.
Slide 4

Deux outils, deux promesses

Graphify

Positionnement : knowledge graph et navigation dans la codebase.

  • Retrouver les zones pertinentes.
  • Relier code, symboles et contexte projet.
  • Améliorer la qualité du handoff développeur.

Code Review Graph

Positionnement : review, impact analysis et blast radius.

  • Identifier ce qui est touché par un changement.
  • Explorer dépendances et relations.
  • Servir de contexte local pour l’agent.
Disponibilité d’un graphe ≠ utilisation optimale du graphe. Le benchmark juge surtout la réponse finale.
Slide 5

Méthodologie benchmark

4 modes comparés

  • base — agent seul
  • graphify — agent + Graphify default
  • crg — agent + Code Review Graph out-of-box
  • graphify+crg — combinaison des deux

6 cas nopCommerce

Everyday, 70 % : admin product edit field, admin permissions, inactive customers CSV.

Hard graph-friendly, 30 % : auth discovery, order total impact, cart-to-order flow.

Qualitéprioritaire

Correctness, coverage, evidence, focus, actionability.

Coûtsecondaire

Prix modèle estimé via summary d’exécution.

Tempsprompt

Le setup est séparé du temps d’exécution.

Setupcaveat

Cold-start répété par sandbox, pas steady-state.
