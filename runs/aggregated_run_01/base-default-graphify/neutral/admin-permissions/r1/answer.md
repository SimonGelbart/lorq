Admin controller authorization is a two-layer check:

1. **Admin-area access**
   Built-in admin controllers inherit [BaseAdminController](/home/simon/repos/lorq-worktrees/base-default-graphify__neutral__admin-permissions__r1/src/Presentation/Nop.Web/Areas/Admin/Controllers/BaseAdminController.cs:12), which applies `[Area(ADMIN)]`, antiforgery, IP validation, `[AuthorizeAdmin]`, vendor validation, and tab-saving filters. `AuthorizeAdminAttribute` is a `TypeFilterAttribute` that resolves `IPermissionService` and checks `StandardPermission.Security.ACCESS_ADMIN_PANEL`; failure returns `ChallengeResult` [AuthorizeAdminAttribute.cs](/home/simon/repos/lorq-worktrees/base-default-graphify__neutral__admin-permissions__r1/src/Presentation/Nop.Web.Framework/Mvc/Filters/AuthorizeAdminAttribute.cs:11), [line 89](/home/simon/repos/lorq-worktrees/base-default-graphify__neutral__admin-permissions__r1/src/Presentation/Nop.Web.Framework/Mvc/Filters/AuthorizeAdminAttribute.cs:89).

2. **Action-specific permissions**
   Actions are guarded with `[CheckPermission(...)]`. Example: `SmsController : BaseAdminController` uses `StandardPermission.Configuration.MANAGE_SMS_SETTINGS` on `Providers` and `MarkAsPrimaryProvider` [SmsController.cs](/home/simon/repos/lorq-worktrees/base-default-graphify__neutral__admin-permissions__r1/src/Presentation/Nop.Web/Areas/Admin/Controllers/SmsController.cs:45). Plugin admin controllers that do not inherit `BaseAdminController` typically add the admin attributes themselves; `PollAdminController` declares `[Area]`, `[AuthorizeAdmin]`, etc., then uses both standard and plugin permissions [PollAdminController.cs](/home/simon/repos/lorq-worktrees/base-default-graphify__neutral__admin-permissions__r1/src/Plugins/Nop.Plugin.Misc.Polls/Admin/Controllers/PollAdminController.cs:22), [line 111](/home/simon/repos/lorq-worktrees/base-default-graphify__neutral__admin-permissions__r1/src/Plugins/Nop.Plugin.Misc.Polls/Admin/Controllers/PollAdminController.cs:111).
